## 2026-06-21 — deploy gotchas
- Caddy SPA + API: do NOT use `try_files {path} /index.html` alongside a `@api` reverse_proxy matcher. Caddy default directive order runs try_files (rewrite) BEFORE reverse_proxy, so /api/* gets rewritten to /index.html and the API returns the HTML page (GET) or 405 (POST). Fix: split into mutually-exclusive `handle /api/*` and `handle` blocks.
- Ubuntu 24.04 `python3 -m venv` fails with "ensurepip is not available" until `apt install python3.12-venv`.
- Windows Git Bash has ssh but NO rsync; use tar-over-ssh. Also ssh known_hosts writes fail when local disk is full — pass `-o UserKnownHostsFile=/dev/null -o StrictHostKeyChecking=no` for unattended runs.
- VPS root used password auth only (no key). Installed Windows id_ed25519.pub into authorized_keys for passwordless deploy.

## [2026-06-20] Inspector w='—' on Trace Core Protocol node — NOT a bug
- Verified the w computation: server-side project_protocol_to_factors() in app/engine/server.py uses the IDENTICAL formula to the client factorWeights(): w(f)=Σ_c dist[c]·support(f→c), normalized to the max non-outcome factor; the 3 outcome nodes (market_state / official_price_transaction_measurement / price_transaction_outcome) = 1.0. Engine values match client (financing_pressure 1.0, borrowing_capacity_subdrivers 0.88, ...).
- The 'trace_core_protocol' node is NOT in factor_weights / VISIBLE_FACTOR_IDS — it is the causal-framework / RUN-PROTOCOL action node (rendered on canvas as the navy CTA button, no w). support={}, weightReady:false → inspector shows '—', which is CORRECT and matches the graph. Do NOT fill a value there (factorWeights would return 1 only as an outcome-category normalization artifact, never displayed). Real factor nodes already show their w.
- ROOT CAUSE of the 'empty engine output': the running uvicorn (--reload) had NOT picked up Codex's server.py edit that added factor_weights/factor_support to /api/run-default-tier — so the API returned no factor_weights and the client silently fell back to its own factorWeights (same values). FIX: restart the uvicorn server (its --reload missed the change). Gotcha: on Windows the uvicorn --reload watcher can silently miss edits — restart manually after backend changes.
- run-default-tier does NOT re-weight scenario_distribution by personal_variables — it returns the public/base distribution regardless of the payload. So the personal conditioning MUST happen client-side: feed the engine distribution as the `base` prior into conditionDistribution(applied, appliedActive, protocolDist). Previously the client used the raw engine dist after a run (protocolDist ? {w:protocolDist} : ...), which made every preset produce identical advice. Fixed to always condition on top.
- Running `bash deploy/deploy.sh` from PowerShell invokes WSL (Linux) bash, NOT Git Bash — paths show as /mnt/c/... The Windows-installed node_modules has the win32 rollup binary, so the Linux build fails with "Cannot find module @rollup/rollup-linux-x64-gnu". Fix: use the native PowerShell deploy `deploy/deploy.ps1` (npm + tar.exe + ssh.exe + scp.exe, no bash). `deploy.sh` still works but ONLY from the actual Git Bash app. Also: PowerShell binary pipelines corrupt tar streams, so deploy.ps1 writes a temp .tgz and scp's it (scp from inside %TEMP% with a bare filename to avoid scp parsing "C:" as a host).
- Deployed site showed "OFFLINE · FALLBACK" (wrong/simpler component set) while local was fine. Root cause: the FIRST deploy built via `cmd //c "npm run build"` did NOT propagate the `VITE_TRACE_API_URL=/api` env var, so that bundle baked the dev default `http://127.0.0.1:8000`. On the HTTPS site those API calls are blocked as mixed content -> fetchFactorTree fails -> TraceWorkspace falls back to FALLBACK_FACTORS/EDGES. The browser had cached that stale index.html+bundle. Fixes: (1) verify the baked base in dist JS (`grep hh=\"...\"` -> must be `/api`, and zero `127.0.0.1`); (2) Caddy now sends `Cache-Control: no-cache` on index.html; (3) deploy scripts `rm -rf dist` before extract so old hashed bundles do not pile up. Always hard-refresh after a deploy when debugging client state.
- CORRECTION to the OFFLINE·FALLBACK entry above: the real root cause was NOT mixed-content. api.ts endpoint paths already include "/api/" (e.g. `${API_BASE_URL}/api/factor-tree`), so building with VITE_TRACE_API_URL=/api produced doubled `/api/api/factor-tree` -> 404 -> fetch fails -> offline fallback. Confirmed via Chrome devtools network: browser requested https://.../api/api/factor-tree 404. Fix: api.ts base now `import.meta.env.VITE_TRACE_API_URL ?? (import.meta.env.DEV ? "http://127.0.0.1:8000" : "")` — same-origin "" in prod, localhost in dev; removed VITE_TRACE_API_URL from both deploy scripts. Verified live: /api/factor-tree + /api/run-default-tier now 200, UI shows full engine-live graph. Lesson: when base + path are concatenated, decide ONCE whether the "/api" prefix lives in the base or the path, never both. Verify the ACTUAL browser request URL (devtools), not just curl against the route you assume.
